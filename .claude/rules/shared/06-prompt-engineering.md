# 提示词写作规范

> **一句话核心：写提示词就像给聪明同事布置任务——说清目标和边界，别堆规则手册。**

---

## 为什么目标导向比规则堆叠更好

### 规则堆叠的问题

想象你给同事布置任务，你会怎么说？

**❌ 规则堆叠版：**
```
你要写一篇文章。
规则1：不要用被动语态
规则2：不要用"的"字超过3次
规则3：不要用"然后"
规则4：不要用"非常"
...（还有16条）
```

你的同事会怎么想？"这是在写文章还是在玩扫雷？"

**✅ 目标导向版：**
```
写一篇给开发者看的技术博客，介绍我们的新功能。
目标读者：有3年经验的前端工程师，他们关心性能和开发体验。
核心信息：这个功能能让他们少写30%的代码，构建速度快2倍。
语气：像技术分享会上的演讲，专业但不端着，可以开点玩笑。
```

### 为什么目标导向更有效

1. **AI 理解目标，不是记规则** — 它通过理解意图来生成内容，给它清晰目标，它会自己找到合适路径
2. **规则越多，冲突越多** — "要简洁"+"要详细"，到底听谁的？规则之间的优先级 AI 猜不准
3. **目标导向更灵活** — 规则是死的，场景是活的，AI 可以根据上下文调整

**一个比喻：**
- 规则堆叠 = 给司机一张"禁止事项清单"
- 目标导向 = 告诉司机"安全平稳地把乘客送到机场，别让他们晕车"

## 怎么做：提示词的结构与写法

### 三段式结构（推荐）

```
【角色定义】你是谁，擅长什么
【任务目标】要做什么，为什么做，给谁看
【输出契约】格式、边界、验收标准
```

#### 1. 角色定义（可选，但有用）

不是"你是一个专业的..."这种废话，而是给 AI 一个视角：

```
你是一个技术博客作者，读者是前端工程师。
你的风格：专业但不端着，会用比喻，偶尔开玩笑。
```

#### 2. 任务目标（核心）

说清楚三件事：做什么、为什么、给谁看：

```
写一篇技术博客，介绍我们的新构建工具。
目标：让前端工程师理解它比 Webpack 快在哪，愿意在项目里试用。
读者：有3年经验，用过 Webpack/Vite，关心性能和开发体验。
```

#### 3. 输出契约（边界）

不是"不要这样不要那样"，而是"输出应该长什么样"：

```
输出格式：标题 + 3段正文 + 1个代码示例
边界：不涉及服务端配置，重点在性能提升
验收标准：读完能理解核心优势，代码示例能复制粘贴直接用
```

### 分层结构（适合复杂场景）

```
System 层（稳定前缀，利于 Cache）
  → 角色定义、通用规范、项目背景、术语表

User 层（任务输入）
  → 当前任务、输入素材、特殊要求

Assistant 层（示例输出，可选）
  → 只在需要 few-shot 时使用
```

### 不同场景的模板

#### 创作类（小说、文案、剧本）

```
【角色】你是[类型]作者，擅长[风格]
【目标】创作[内容]，给[读者]看，达成[效果]
【素材】[背景/人物/情节]
【风格】[参考作品/具体描述]
【边界】[不要的元素/长度限制]
【验收】[读完的感觉/具体指标]
```

#### 提取类（总结、分析、整理）

```
【任务】从[来源]提取[信息]
【目的】用于[场景]，给[对象]看
【重点】关注[维度]，忽略[维度]
【格式】[结构/字数/样式]
【示例】[期望输出的样子]
```

#### 分析类（评估、对比、建议）

```
【背景】[项目/问题/现状]
【目标】分析[对象]，得出[结论]
【维度】从[角度1/角度2/角度3]评估
【标准】[好的定义/坏的定义]
【输出】[结论+理由+建议]
```

## 踩坑案例：好坏对比

### 坑1：规则手册 → 目标描述

**❌ 坏：**
```
不要用"然而"、不要用"值得注意的是"、不要用"总的来说"...（20条）
```

**✅ 好：**
```
写得像人在聊天，不要像论文。
参考风格：知乎高赞回答、B站up主的文案。
避免：学术腔、翻译腔、说教腔。
```

**教训：** 规则越多，AI 越像在"规避检测"而不是在"创作"。

---

### 坑2：强制语气 → 风格参考

**❌ 坏：**
```
你必须用轻松活泼的语气！一定要幽默！千万不要严肃！
```

**✅ 好：**
```
语气：像朋友推荐好东西，专业但不端着。
参考：李诞的脱口秀（有梗但不油腻）、罗翔的普法视频（严肃话题轻松讲）。
```

**教训：** 强制语气会让 AI 用力过猛，反而尴尬。

---

### 坑3：格式说明书 → 示例输出

**❌ 坏：**
```
输出格式要求：
1. 第一行是标题，标题要简洁有力，不超过20字
2. 第二行空一行
3. 第三行开始正文，正文分3段
4. 每段150-200字
...（10条）
```

**✅ 好：**
```
输出格式参考：

# 这个功能让你少写30%的代码

过去你要写一堆配置文件，现在只需要一行命令...

怎么做到的？核心是智能推断...

试试看？安装只要10秒...
```

**教训：** 一个好示例胜过100字格式说明。

---

### 坑4：重复注入 → 一次说清

**❌ 坏：**
```
你要写得专业。记住，一定要专业。我再强调一遍，专业很重要。千万不要忘记，要保持专业。
```

**✅ 好：**
```
语气：技术分享会上的演讲，专业但不端着。
术语要准确，但要解释清楚，让非专家也能听懂。
```

**教训：** 重复不会让 AI 更听话，只会浪费 token。

---

### 坑5：模糊要求 → 验收标准

**❌ 坏：**
```
写一篇好文章。要有吸引力。要让人想读。
```

**✅ 好：**
```
写一篇技术博客，介绍新功能。
验收标准：
- 读完能说出3个核心优势
- 知道怎么开始用（有具体步骤）
- 愿意在项目里试试（不是"听起来不错"而是"我要试试"）
```

**教训：** "好"是什么？谁说了算？要给可验证的标准。

## Cache 优化：稳定前缀的设计

### 为什么要优化 Cache

大模型 API 的 prefix cache 机制：重复的输入内容可以缓存，价格便宜很多。

**实测数据（来自掠蓝写作 Pro）：**
- 优化前：Cache 命中率 10%，月成本 $150
- 优化后：Cache 命中率 80%，月成本 $40
- **节省 75% 成本，首 token 时间从 2-3 秒降到 0.3-0.5 秒**

### Cache 友好的布局原则

**核心思路：稳定的放前面，变化的放后面**

```
【稳定前缀 - 会被缓存】
- 角色定义
- 项目背景
- 通用规范
- 术语表
- 风格指南

【变化部分 - 每次不同】
- 当前任务
- 输入素材
- 特殊要求
```

**❌ 不利于 Cache：**
```
当前任务：续写第3章
[具体内容]

你是掠蓝写作 Pro 的内容生成引擎。
项目背景：...
通用规范：...
```

**✅ 利于 Cache：**
```
【System - 稳定前缀】
你是掠蓝写作 Pro 的内容生成引擎。
项目背景：...
通用规范：...
风格指南：...

【User - 变化部分】
当前任务：续写第3章
[具体内容]
```

### 三层分层策略

| 层级 | 内容 | 更新频率 |
|------|------|----------|
| 第一层：全局稳定 | 角色定义、项目背景、基础规范 | 几周到几个月 |
| 第二层：场景稳定 | 任务类型说明、该类任务通用要求 | 几天到几周 |
| 第三层：任务变化 | 具体任务、输入素材、特殊要求 | 每次调用 |

### 注意事项

- 不要为了 Cache 牺牲质量，关键信息就放稳定前缀
- `stylePrompt` 如果是用户临时改的自由文本，放前面会打碎 Cache，应后置
- 稳定前缀确定后就不要频繁改动，要有版本管理

## 不同模型的特点与针对性写法

### GPT-5.4（s2api）

**擅长：** 代码实现、技术方案、深度探查、细节排查、测试用例

**不擅长：** 文案写作（容易写成说明书）、前端审美、大局观

**Prompt 侧重：** 明确任务边界、验收标准、技术细节

```
任务：重构用户认证模块
当前状态：认证逻辑分散在3个文件里，没有统一错误处理
目标：统一到 auth.js，添加完整错误处理，覆盖率达80%
不要改：数据库 schema、API 路由定义、前端调用方式
验收标准：所有现有功能正常，新增测试全部通过
```

---

### Claude Sonnet 4.5（dome）

**擅长：** 产品文案、用户引导、设计思考、方案评估、审查 GPT 的文案产出

**不擅长：** 细节容易粗心，长时间执行容易偷懒

**Prompt 侧重：** 目标效果、用户感受、风格参考

```
任务：写产品介绍文案
目标读者：中文网文作者，担心 AI 味被识别
核心价值：场景黑箱调模、默认去 AI 味、国内外平台合规
语气：像朋友推荐好东西，专业但不端着
验收标准：读完能说出3个核心优势，感觉"这个工具懂我"
```

---

### Kimi K2.6（cline）

**擅长：** 前端 UI 审查、视觉设计评估、响应式设计检查、中短任务前端实现

**不擅长：** 长任务容易出错，后端逻辑不如 GPT

**Prompt 侧重：** 视觉效果、交互体验、设备适配

```
任务：审查写作台页面的移动端适配
评估维度：布局合理性、按钮大小（至少44×44px）、文字清晰度（至少14px）
参考：飞书文档的移动端、Notion 的移动端
输出：发现的问题（按优先级排序）+ 具体改进建议（能直接实施）
```

---

### 多模型协作策略

**功能实现 + 文案审查：**
```
1. GPT 实现功能（包括 UI 文案、错误提示等）
2. Sonnet 审查文案（不能随意通过，要用自己的判断标准）
3. GPT 根据反馈修改
```

**设计评估 + 实现 + 审查：**
```
1. Sonnet 提供设计思路和大局观
2. Kimi 评估视觉效果和前端可行性
3. GPT 实现功能
4. Kimi 审查前端视觉
5. Sonnet 审查整体体验
```

**交叉验证：**
```
同一个问题让多个模型分别分析：
- GPT 找细节问题
- Sonnet 看大局和用户体验
- Kimi 看前端视觉
主模型综合多方结果做最终判断
```

| 模型 | 侧重点 | 关键词 |
|------|--------|--------|
| GPT-5.4 | 任务边界、验收标准、技术细节 | 明确、具体、可验证 |
| Sonnet 4.5 | 目标效果、用户感受、风格参考 | 自然、有温度、像人话 |
| Kimi K2.6 | 视觉效果、交互体验、设备适配 | 直观、顺畅、好看 |

## 中文创作场景的特殊考虑

### 常见翻译陷阱

**直译"Use active voice"：**
- ❌ `使用主动语态，不要使用被动语态。`
- ✅ `句式：多用"谁做了什么"，少用"什么被怎么样"。`

**直译"Be concise"：**
- ❌ `要简洁。`
- ✅ `用短句，不要长句套长句。用白话，不要文言文。`

**直译"Professional tone"：**
- ❌ `使用专业的语气。`
- ✅ `语气：像技术分享会上的演讲，专业但不端着。术语要准确，但要解释清楚。`

### 要避免的中文腔调

| 类型 | 坏例子 | 好例子 |
|------|--------|--------|
| 翻译腔 | 该功能实现了对用户体验的优化 | 这个功能让用户用起来更顺手 |
| 说教腔 | 值得注意的是... | 注意... |
| 文言文 | 此功能可优化用户体验 | 这个功能能让用户用起来更顺手 |

### 网文/创作类任务的特殊要求

**平台风格差异：**
- 起点：偏爽文，节奏快，金手指明显
- 晋江：偏情感，人物细腻，注重心理描写
- 番茄：偏短平快，章节短，悬念密集

**去 AI 味的正确方式：**
- ❌ 列20个禁词（让 AI 在"规避检测"）
- ✅ 描述文字气质目标（让 AI 在"创作"）

```
## 文字气质
- 让场面像正在发生：多写动作、对话、感官和即时反应
- 让文字有呼吸感：句子和段落自然伸缩，快处利落，重场面肯停下来写
- 让人物像活人：说话带脾气，小动作、迟疑、误判都可以留下
- 让推进可被看见：关系、阵营、位置变化要让读者能从行为结果看出来
```

## 实战检查清单

写完 Prompt 后，用这个清单自查：

### 目标清晰度
- [ ] 说清楚了要做什么？
- [ ] 说清楚了为什么做？
- [ ] 说清楚了给谁看？
- [ ] 说清楚了验收标准？

### 反模式检查
- [ ] 没有堆砌"不要用"的规则列表？
- [ ] 没有重复强调同一件事？
- [ ] 没有用模糊形容词代替具体描述（"好"、"专业"、"有吸引力"）？
- [ ] 没有写成格式说明书？

### Cache 友好性
- [ ] 稳定内容放在最前面？
- [ ] 动态内容放在最后面？
- [ ] 避免了在稳定区域插入动态内容？

### 快速自测：三个问题

1. **如果你是 AI，看完这个 prompt，知道要做什么吗？** → 不知道说明目标不清晰
2. **如果你是 AI，看完这个 prompt，知道怎么做吗？** → 不知道说明上下文不足
3. **如果你是 AI，做完之后，知道怎么验收吗？** → 不知道说明验收标准不明确
